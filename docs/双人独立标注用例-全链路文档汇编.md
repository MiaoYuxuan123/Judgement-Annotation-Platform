### 核心用例3：裁定最终标注结果
| 项目 | 内容 |
| ---- | ---- |
| 用例名称 | 裁定最终标注结果 |
| 参与者 | 裁定者 |

| 后置条件 | 最终裁定版本被锁定，平台内数据不可修改，可导出。标注员只能查看最终结果，不可查看对方中间标注。 |

### 1.3 定义、缩写

- **SRS**：Software Requirements Specification
- **命题**：从裁判文书中抽取的具有独立语义的语句单元
- **一级标签**：SF/GF/SM/GM 四类命题性质判断
- **二级标签**：GM 下的法律、解释、合同、习惯、道德、其他规范
- **关系**：支持 S、反对 A、组合 J、匹配 M、同一 I
- **四阶段流程**：创建任务 → 标注 → 冲突解决 → 结果输出（不可逆）
- **裁定者**：单个标注项目的唯一终审人，一次裁定生效
- **序号自动前移**：删除某命题后，后续序号依次向上补齐

### 2.3 用户特征（权责分离）

1. **超级管理员**：用户管理、角色分配、权限配置、全局设置
2. **任务创建者**：上传文档、配置标签、分配标注员、指定裁定者
3. **标注员**：仅完成自身被分配的文本标注、命题标注、关系标注
4. **裁定者**：查看双标注结果、人工比对差异、编辑并确认最终版本

### 2.4 约束

- 仅处理纯文本，不处理图片、扫描件
- 任务启动后**标签体系不可修改**
- 命题序号**按文本出现顺序自动生成**，删除则后续自动前移
- 无撤销 / 重做，仅支持**删除→重新创建**
- 四阶段流程**不可逆**，导出后平台数据不可修改
- 单个标注项目**仅一名裁定者、一次裁定生效**
- 标注员不可查看他人中间标注，仅可查看最终结果
- 系统仅保留**最后一次提交**的标注结果

#### 3.2.7 系统特性 7：协同标注与冲突裁定

##### 3.2.7.1 特性描述

支持多用户独立标注，裁定者可人工比对双标注结果并裁定最终版本。

##### 3.2.7.2 刺激 / 响应序列

1. 刺激：管理员创建任务并分配标注员；响应：系统为每个标注员生成独立的标注页面
2. 刺激：两名标注员完成同一文档的标注；响应：系统将两份标注结果提交给指定裁定者
3. 刺激：裁定者查看双标注结果；响应：系统并排展示两份标注，清晰呈现差异
4. 刺激：裁定者修改标注内容或采纳某一方；响应：系统保存最终版本，一次裁定即为最终结果

##### 3.2.7.3 相关功能需求

- **FR-029** 支持多用户登录，角色权责分离，权限由超级管理员分配。**优先级：Must｜来源：[确认规则]**
- **FR-030** 一份文档分配给至少 2 人独立标注，互不干扰。**优先级：Must｜来源：[人工补充]**
- **FR-031** 系统不自动比对差异，仅并排展示两人标注结果。**优先级：Must｜来源：[确认规则]**
- **FR-032** 裁定页面展示原文、双标注结果，清晰呈现差异。**优先级：Must｜来源：[人工补充]**
- **FR-033** 裁定者可直接修改标注内容，一次裁定即为最终版。**优先级：Must｜来源：[确认规则]**
- **FR-034** 大任务可设多名裁定者，但单个项目仅绑定一名裁定者。**优先级：Must｜来源：[确认规则]**

## ADR-001：采用模块化垂直架构 (Modular Vertical Architecture)

### 状态

已接受

### 背景

我们团队此前从未有过Web开发的经验，且该项目开发周期紧迫。若采用传统的水平分层架构，多人频繁修改同一代码模块可能导致严峻的 Git 合并冲突。此外，由于成员对业务边界认知可能存在偏差，极易出现“协作阻塞”现象，增加项目交付风险。

### 决策

我们决定采用模块化垂直架构。将系统按业务功能进行垂直切分，目前暂定是用户管理、任务管理、标注界面、裁定界面这四部分。每位成员负责一个完整模块的端到端开发，包括前端页面、后端逻辑及数据库表设计，模块间通过预定义的 API 接口进行轻量化交互。

### 理由

- **低耦合协作**：垂直切分确保了代码物理隔离，极大减少了多人协作时的 Git 冲突成本。
- **并行效率最大化**：成员拥有独立的业务空间，无需等待他人完成底层通用类即可推进业务。
- **责任明确**：每个模块有唯一的负责人，便于 Bug 追踪和课程作业的个人绩效评定。

### 后果

- **正面影响**：开发并行度高，代码合并冲突减少，交付压力得到缓解。
- **负面影响**：可能存在少量公共工具类的重复实现；模块功能划分需要前期仔细商讨；对前期 API 契约设计的准确性要求较高。
- **需要关注的风险**：必须在初期强制统一 API 返回格式和全局异常处理，否则后期联调会面临巨大重构压力。

### AI 辅助记录

- **AI 初稿内容摘要**：建议使用经典 MVC 三层架构，侧重于代码复用和逻辑分层，未考虑团队协作冲突。
- **人工修订内容**：改为“模块化垂直架构”，强调按业务功能切分以匹配四人团队的协作模式，以此提高开发效率。
- **修订理由**：AI 忽略了开发效率这一关键质量属性在短工期项目中的优先级。垂直架构虽牺牲了部分复用性，但换取了极高的协作并行度，可以极大减少因认知偏差造成的项目进度拖延。

## ADR-002：采用 Vue3 + Spring Boot 前后端分离架构

### **状态**

已接受

### **背景**

项目需要开发一个交互复杂的标注系统，涉及动态表单、实时预览等功能。我们团队成员具备Java和基础前端能力，技术选型需考虑当前的主流趋势和开发效率。项目需在本地服务器或云主机部署，且需要快速迭代。

### **决策**

我们决定采用 Vue3 作为前端框架，配合 Spring Boot 作为后端框架，通过 RESTful API 进行数据交互。构建工具使用 Vite，UI组件库选用 Element Plus。

### **理由**

- **技术成熟度**：Spring Boot 是 Java 领域最成熟的后端框架，生态极其完善；Vue3 已成为前端通用工具，技术成熟稳定。
- **开发体验**：Vite构建工具启动速度极快，热更新无延迟，极大提升学生开发体验。
- **生态丰富**：两者都有海量的开源组件和文档，遇到Bug能快速找到解决方案，降低课程作业的失败风险。
- **前后端分离**：符合现代Web开发标准，便于前后端并行开发，且界面美观度高，利于课程答辩展示。

### **后果**

- **正面影响**：开发效率高，界面交互流畅，系统扩展性好。
- **负面影响**：增加了跨域配置和静态资源部署的复杂度；首屏加载需优化。
- **需要关注的风险**：需严格管理 Node.js 和 Java 的版本依赖，避免因环境差异导致系统不兼容的问题。

### 5.3 CI/CD 流水线

> **来源：** `docs/P4/CICD 配置与运行记录文档.md` 第 7–20 行

## 一、CI/CD 整体说明

本项目基于 **GitLab CI/CD** 搭建流水线，采用**父子流水线拆分**设计：

- 根流水线：`.gitlab-ci.yml`
- 后端子流水线：`.gitlab/backend.yml`
- 前端子流水线：`.gitlab/frontend.yml`

流水线逻辑：

1. 检测代码提交改动目录
2. 只改动后端：自动触发后端构建、检查、测试流程
3. 只改动前端：自动触发前端构建、检查、测试流程
4. 仅修改文档、Bug 日志、README、CI 配置等**非前后端代码**时，执行空任务占位，避免流水线为空失败

---

> **来源：** `docs/P4/CICD 配置与运行记录文档.md` 第 22–64 行

## 二、根流水线配置文件内容（.gitlab-ci.yml）

```yaml
stages:
  - trigger

# 触发后端 pipeline
backend:
  stage: trigger
  trigger:
    include:
      - local: .gitlab/backend.yml
    strategy: depend
  rules:
    - changes:
        - backend/**/*

# 触发前端 pipeline
frontend:
  stage: trigger
  trigger:
    include:
      - local: .gitlab/frontend.yml
    strategy: depend
  rules:
    - changes:
        - frontend/**/*

# 无前后端变更时空任务兜底
noop:
  stage: trigger
  image: alpine:3.20
  rules:
    - changes:
        - backend/**/*
      when: never
    - changes:
        - frontend/**/*
      when: never
    - when: on_success
  script:
    - echo "No backend/frontend changes; skip build triggers."
```
