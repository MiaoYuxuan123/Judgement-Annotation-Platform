## 1 引言

### 1.1 目的

本文档详细描述了法律裁判文书标注平台（Judgement Annotation Platform, JAP）的软件需求规格。该平台旨在为法学院师生提供一个协作式裁判文书论证结构标注环境，支持多角色协同标注、冲突裁决及论证结构可视化分析。本文档面向项目经理、开发团队、测试团队、法学院教授及学生用户，作为系统设计、开发、测试和验收的基准依据。

### 1.2 文档约定

**优先级标记：**

- [P0]：核心需求，必须在首个发布版本中实现
- [P1]：重要需求，应在首个版本中实现
- [P2]：增强需求，可在后续迭代中实现

**术语约定：**

- "系统"指代本标注平台
- "文书"指代裁判文书原文
- "标注"指代对论证要素（主张、理由、依据、结论等）的标记行为

**格式规范：** 本文档遵循IEEE Std 830-1998推荐实践标准。

### 1.4 项目范围

**目标：** 构建面向法学教育与研究领域的裁判文书结构化标注系统，实现论证要素的精细化标记、多人协作标注的冲突解决机制，以及论证结构的可视化呈现。

**主要功能范围：**

- 多角色任务分配与管理（管理员、标注者、裁定者）
- 支持中文裁判文书的论证结构标注（包括主张、事实、理由、法律依据、结论等要素）
- 多人标注冲突检测与裁决工作流
- 标注数据的多格式导出（JSON, CSV, XML）
- 论证结构自动图谱生成（支持力证、反驳、补强等关系）

**排除范围：** 系统不包含裁判文书的自动爬取功能（需手动上传）、不包含自然语言处理自动标注模块（后续版本可扩展）。

## 2 总体描述

### 2.1 产品视角

本系统是一个独立的Web应用，采用B/S架构，服务于法学院内部教学与研究场景。系统架构如下：

| 层级 | 组件 | 技术 |
|------|------|------|
| 表示层 | 用户浏览器 | Vue.js SPA |
| 应用层 | Django REST API | Python 3.10+ |
| 数据层 | PostgreSQL | PostgreSQL 14+ |
| 缓存层 | Redis | Redis 6+ |

### 2.2 产品功能

**核心功能清单：**

1. **任务管理模块 [P0]**：管理员创建、编辑、删除标注任务；上传裁判文书；配置标注schema；分配标注者与裁定者
2. **协同标注模块 [P0]**：标注者认领任务；基于文本的富客户端标注界面；支持实体标注和关系标注；自动保存与版本控制
3. **冲突裁决模块 [P0]**：系统自动检测多人标注冲突；冲突可视化对比视图；裁定者决策与批注；冲突解决历史追踪
4. **数据导出模块 [P1]**：标注数据导出（JSON, CSV, XML）；论证结构图导出（PNG, SVG, PDF）；统计报告生成
5. **可视化分析模块 [P1]**：自动生成论证结构图；支持力证、反驳、补强关系可视化；交互式图谱浏览与筛选

### 2.3 用户类型和特征

| 用户角色 | 描述 | 技术能力 | 使用频率 | 主要需求 |
|----------|------|----------|----------|----------|
| 管理员 | 教授或项目负责研究生 | 中高 | 低频 | 任务配置、权限管理、质量控制 |
| 标注者 | 参与课程的本科生/研究生 | 中 | 高频 | 直观标注界面、效率工具、进度查看 |
| 裁定者 | 高年级研究生或教授 | 高 | 中频 | 冲突对比视图、决策支持、批注工具 |
| 访客 | 仅浏览的旁听学生 | 低 | 低频 | 只读访问、示例查看 |

### 2.5 设计和实现约束

**技术约束：**

- 必须基于Django + Vue.js技术栈（团队现有技术积累）
- 数据库必须使用PostgreSQL（支持JSON字段存储复杂标注数据）

**业务约束：**

- 标注schema必须兼容主流法学论证理论（图尔敏模型等）

- 依赖Django-import-export库实现数据导出功能
- 依赖D3.js或ECharts实现论证结构可视化

## 3 具体需求

### 3.1 功能需求

#### 3.1.1 管理员功能需求

**FR-ADM-001：任务创建 [P0]**

管理员应能创建新的标注任务。输入：任务名称、任务描述、截止日期、上传的裁判文书文件。处理：系统生成任务ID，存储文书内容至数据库。输出：任务创建成功确认，任务状态设为"待分配"。

**FR-ADM-002：Schema配置 [P0]**

配置该任务支持的标注类型与关系类型。输入：实体类型列表（如：主张、事实、理由、法条、结论）、关系类型列表（如：支持、反驳、补强）、颜色编码。处理：系统验证schema完整性（至少包含一个实体类型）。输出：Schema绑定至特定任务，标注者只能使用已配置的类型。

**FR-ADM-003：人员分配 [P0]**

为任务分配标注者和裁定者。输入：选择用户（多选）、指定每份文书的标注人数（>=2，用于冲突检测）、指定裁定者。处理：系统生成标注分配记录，发送通知。输出：分配确认，任务状态更新为"进行中"。

**FR-ADM-004：进度监控 [P1]**

查看任务整体进度与个体标注质量。输入：任务ID查询。处理：统计各标注者完成度、计算标注一致性率（Cohen's Kappa）。输出：仪表盘显示进度条、一致性热力图、导出统计报告。

#### 3.1.2 标注者功能需求

**FR-ANN-001：任务接收 [P0]**

查看并认领/接受分配的任务。输入：用户ID验证。处理：系统检索该用户的待办任务列表。输出：显示任务列表（包含优先级、截止日期、完成状态）。

**FR-ANN-002：文本标注 [P0]**

在裁判文书文本上执行Span标注和Relation标注。输入：鼠标选中文本、选择实体类型、添加标签注释。处理：记录实体位置（开始偏移量、结束偏移量）、类型、颜色；支持重叠标注（同一文本可被多次标注不同类型）；自动保存草稿（每30秒或手动触发）。输出：标注实体高亮显示，右侧边栏显示标注列表。

**FR-ANN-003：关系建立 [P0]**

建立标注实体间的论证关系。输入：选择源实体、目标实体、选择关系类型（支持、反驳、补强等）。处理：验证关系合法性（如：禁止自环）、存储关系数据。输出：可视化连线显示（不同颜色/线型表示不同关系）。

**FR-ANN-004：标注提交 [P0]**

完成标注后提交。输入：提交确认。处理：系统锁定该用户对此任务的编辑权限，触发冲突检测流程。输出：提交确认，状态更新为"待裁决"。

**FR-ANN-005：标注修改 [P1]**

在提交前或裁决前修改已有标注。输入：编辑或删除操作。处理：版本控制保留历史，支持撤销/重做（Undo/Redo栈深度>=10）。输出：实时更新界面显示。

#### 3.1.3 裁定者功能需求

**FR-ARB-001：冲突查看 [P0]**

查看系统检测到的标注冲突。输入：任务ID，冲突筛选条件（类型：实体冲突/关系冲突/缺失冲突）。处理：系统比对同一文书的多个标注版本，识别：实体边界不一致（重叠但不相等）、实体类型分歧（同一文本标注为不同类型）、关系连接差异。输出：分屏对比视图（左/中/右列显示不同标注者视角）。

**FR-ARB-002：冲突裁决 [P0]**

对冲突项做出裁定决策。输入：选择采纳哪个标注者的方案，或输入新的修正方案，添加裁决理由。处理：记录裁决结果，标记冲突状态为"已解决"。输出：生成黄金标准（Gold Standard）标注数据。

**FR-ARB-003：批量裁决 [P1]**

对相似冲突进行批量处理。输入：选择多个同类型冲突，应用统一决策规则。处理：系统应用规则，生成批量裁决记录。输出：更新后的黄金标准数据。

#### 3.1.4 数据导出与可视化需求

**FR-EXP-001：数据导出 [P1]**

导出标注结果。输入：选择任务、选择导出格式（JSON/CSV/XML）、选择数据版本（原始标注/裁决后）。处理：格式化数据，包含：元数据（任务信息、标注者、时间戳）、实体列表（ID、文本、位置、类型）、关系列表（源ID、目标ID、关系类型）。输出：下载文件，JSON Schema需符合JSON-LD标准。

**FR-VIZ-001：论证结构图生成 [P1]**

自动生成论证结构可视化图表。输入：标注数据（实体与关系）。处理：力导向图布局（Force-directed layout）；节点按类型着色，边按关系类型定线型；支持力证（实线绿色）、反驳（虚线红色）、补强（点线蓝色）。输出：交互式SVG图表，支持缩放、拖拽、点击查看详情。

**FR-VIZ-002：图谱导出 [P2]**

将生成的论证结构图导出为静态文件。输入：选择导出格式（PNG/SVG/PDF）、分辨率设置。处理：渲染图表，转换格式。输出：图片文件下载。

### 3.2 外部接口需求

#### 3.2.1 用户界面

**UI-001：标注界面布局 [P0]**

- 左侧：文书文本区（占60%宽度），支持行号显示、段落折叠
- 右侧：操作面板（占40%宽度），包含实体类型选择器、当前标注列表、关系建立工具、提交按钮

**UI-002：冲突裁决界面 [P0]**

- 采用三栏布局：左栏显示标注者A的标注视图、中栏显示标注者B的标注视图、右栏为裁决操作区

**UI-003：响应式设计 [P1]**

- 支持1280px以上宽度屏幕的完整功能显示
- 移动端（<768px）仅支持查看功能，不支持标注操作

#### 3.2.2 软件接口

**SI-001：RESTful API [P0]**

遵循REST架构，JSON数据格式。主要端点：

- GET /api/tasks/ - 获取任务列表
- POST /api/tasks/{id}/annotations/ - 创建标注
- PUT /api/annotations/{id}/ - 更新标注
- GET /api/tasks/{id}/conflicts/ - 获取冲突列表
- POST /api/conflicts/{id}/resolve/ - 提交裁决

**SI-002：认证接口 [P0]**

- 基于JWT（JSON Web Token）的身份验证
- 支持Token刷新机制（Refresh Token）
- 会话有效期：8小时

#### 3.2.3 通信接口

**CI-001：HTTP/HTTPS [P0]**

- 生产环境必须使用HTTPS（TLS 1.2+）
- 支持HTTP/2协议优化多资源加载

### 3.4 设计约束

**DC-001：技术栈约束**

- 后端必须使用Django框架，遵循MTV架构模式

**DC-002：安全约束**

- 所有用户密码必须使用PBKDF2算法哈希存储
- 防范SQL注入（使用Django ORM参数化查询）
- 防范XSS攻击（Vue模板自动转义 + CSP策略）
- 文件上传限制：仅允许.txt, .docx, .pdf，大小<=10MB

**DC-003：可访问性**

- 遵循WCAG 2.1 AA级标准
- 所有图表必须提供替代文本描述
- 支持键盘导航操作

**SEC-001：数据隐私 [P0]**

- 符合《个人信息保护法》要求，不收集不必要的个人信息

### 附录B：数据模型概要

核心实体：

- User：用户（管理员、标注者、裁定者）

- Conflict：冲突记录（类型、位置、涉及标注）

### 附录D：验收标准清单

**功能验收：**

- 管理员可成功创建包含5份文书的任务
- 2名标注者可独立完成同一份文书的标注
- 系统自动识别并显示边界重叠的实体冲突
- 裁定者可采纳A的实体边界+B的实体类型进行裁决
- 导出JSON包含完整的论证结构树

**性能验收：**

- 50个并发用户下，标注操作延迟<1秒
- 导出100份文书的标注数据<10秒

**安全验收：**

- 未登录用户无法访问任何API端点
- 标注者无法查看他人的未完成标注
- 上传.exe文件被系统拒绝

## AI需求盲区挑战对比分析

| 对比维度 | AI生成的需求 | 人工调研发现的需求 | 分析 |
|----------|-------------|-------------------|------|
| **AI正确覆盖的需求** | - 多角色权限分离（管理员/标注者/裁定者）<br>- 任务创建与分配机制<br>- 数据导出功能（JSON/CSV/Excel）<br>- 论证结构可视化与图示生成<br>- 文本导入与裁判理由提取<br>- 冲突解决与裁决流程<br>- 标注数据持久化存储<br>- 基本的非功能需求（性能、兼容性） | - 超级管理员、任务创建者、标注员、裁定者四角色分离<br>- 支持Excel/JSON导出论证结构图<br>- 实时自动生成论证图示<br>- 多人标注冲突由裁定者解决<br>- 任务分阶段管理<br>- 支持PDF/Word/文本导入 | AI基本覆盖了核心业务流程与角色体系，但深度与精确度不足 |
| **AI遗漏的需求** | - 未提及四阶段不可逆流程（创建→标注→冲突解决→结果输出）<br>- 未定义SF/GF/SM/GM多级标签体系<br>- 未规定命题序号自动生成与自动前移机制<br>- 未明确"单个项目仅一名裁定者，一次裁定生效"规则<br>- 未要求任务启动后标签体系锁定<br>- 未规定标注员不可查看他人中间标注<br>- 未明确三栏布局的具体内容定义（左侧原文+命题列表、中间标注区、右侧图示区）<br>- 未包含形式化表达式标注（如S(1,2)）<br>- 未要求系统仅保留最后一次提交结果<br>- 未提及论证图示区仅允许拖拽节点、不可新建/修改关系 | - 四阶段流程：创建任务→标注→冲突解决→结果输出（不可逆，导出后平台数据不可修改）<br>- 标签体系：一级标签SF/GF/SM/GM，GM下含6个二级标签（法律/解释/合同/习惯/道德/其他）<br>- 关系类型：支持S、反对A、组合J、匹配M、同一I（AI仅提支持/反驳）<br>- 命题管理：按文本出现顺序自动编号，删除后后续序号自动前移<br>- 裁定规则：单个标注项目仅绑定一名裁定者，一次裁定即为最终版<br>- 视图隔离：标注员仅查看自己标注与最终结果，不可看他人中间标注<br>- 形式化表达：支持S(1,2)、S({1,2},3)等关系表达式<br>- 平台锁定：导出后平台内数据锁定不可修改 | 遗漏了人工确认的核心业务规则与数据结构定义，尤其是"四阶段不可逆"、"单裁定者终审"、"序号自动前移"等关键约束。AI之所以会遗漏，有相当大的一部分是因为无法知晓客户的个性化需求，另一部分原因是无法知晓作为开发者的我们准备怎么实现这个系统。 |
| **AI捏造的需求** | - 兼容图尔敏模型等法学论证理论<br>- 实体类型定义（主张/事实/理由/法条/结论）<br>- 自动冲突检测与Cohen's Kappa一致性计算<br>- 重叠标注支持<br>- 撤销/重做功能（Undo/Redo栈深度>=10）<br>- 访客角色（仅浏览的旁听学生）<br>- JSON-LD标准导出格式<br>- Docker容器部署需求<br>- 移动端支持（<768px查看功能）<br>- 基于JWT的认证与Token刷新<br>- 自动保存草稿（每30秒）<br>- 批量裁决功能<br>- 支持XML导出格式<br>- 单份文书最大5万字限制<br>- 支持分段折叠与行号显示 | - 无图尔敏模型要求<br>- 无实体类型分层（主张/事实等），而是SF/GF/SM/GM标签体系<br>- 无自动冲突检测算法，明确"系统不自动比对差异，仅并排展示"<br>- 无撤销/重做，明确"不提供撤销/重做，仅支持删除→重新创建"<br>- 无访客角色<br>- 无JWT认证要求<br>- 无自动保存草稿机制<br>- 无批量裁决功能<br>- 仅支持Excel/JSON/图片导出，无XML | AI引入了通用标注系统的常见功能（如撤销重做、自动冲突检测、访客模式），但这些恰恰是人工明确排除的约束。 |
| **AI表述有歧义的需求** | - "协同标注"：描述为通用协同，未明确是"多人独立标注"而非"实时协同编辑"<br>- "冲突检测"：称"系统自动检测冲突"，未明确人工确认"不自动比对，仅并排展示"<br>- "关系类型"：仅列支持/反驳/补强，遗漏J/M/I三种关键关系类型<br>- "版本控制"：声称"版本控制保留历史"，与人工要求"仅保留最后一次提交"矛盾<br>- "裁定者"：称"大任务可设多名裁定者"，与人工"单个项目仅绑定一名裁定者"冲突<br>- "标注修改"：允许"提交前或裁决前修改"，未明确"仅保留最后一次提交"的覆盖机制<br>- "论证结构"：使用通用法学论证术语，未对应到具体的形式化表达式体系 | - 多人独立标注：明确"一份文档分配给至少2人独立标注，互不干扰"（非实时协同）<br>- 冲突展示：明确"系统不自动比对差异，仅并排展示两人标注结果"<br>- 关系体系：明确S（支持）、A（反对）、J（组合）、M（匹配）、I（同一）<br>- 数据覆盖：明确"系统仅保留最后一次提交的标注结果"<br>- 终审机制：明确"裁定者可直接修改标注内容，一次裁定即为最终版"<br>- 标签锁定：明确"任务启动后标签体系不可修改" | AI使用了通用的、含糊的术语，而人工需求有精确的业务规则定义，特别是"独立性"、"一次性"、"不可逆"等强约束。 |
