1. **当前工作流程**
   - 现在如何处理法律文档（案例、法条、论文、合同）？
   - 使用什么工具进行阅读和标注？（纸质、PDF阅读器、Word、Notion等）
   - 现有方案的最大痛点是什么？

2. **用户群体画像**
   - 主要使用者是谁？（本科生、研究生、教师、研究人员比例）
   - 不同角色的使用频率和深度差异？
   - 是否有跨院校协作需求？

### 第二部分：核心功能需求（30分钟）

#### **文档管理维度**
3. **支持的文档类型**
   - 需要支持哪些格式？（PDF、Word、图片扫描件、网页、视频音频？）
   - 文档来源？（北大法宝、Westlaw、裁判文书网、自建数据库？）
   - 单文档体量？（几页到几百页不等？法典全文标注？）

4. **标注类型与规范**
   - 基础标注：高亮、下划线、删除线、批注？

     - 法条引用标注（关联到具体条款）
     - 案例要旨提取
     - 争议焦点标记
     - 证据链标注
     - 时间轴标注（刑事案件时间线）
   - 是否需要结构化标注（如：标签分类 = 程序问题/实体问题/证据问题）？

5. **标注的组织与检索**
   - 如何整理标注？（文件夹、标签、项目、案件维度）
   - 检索需求：按关键词、按标注类型、按法律领域、按引用频率？
   - 是否需要可视化展示？（思维导图、时间线图谱）

#### **协作与共享**
6. **协作场景**
   - 是否需要多人协作标注同一文档？（课题组、模拟法庭、案例研讨）
   - 协作模式：实时同步 vs 异步批注？
   - 权限管理：谁能看、谁能改、谁能批注批注？

7. **知识沉淀**
   - 个人知识库 vs 机构知识库？
   - 优秀标注是否可以被他人引用或学习？
   - 是否支持标注的导出和引用（生成文献综述、案例分析报告）？

### 第三部分：专业场景深挖（25分钟）

#### **教学场景**
8. **课程与案例教学**
   - 教师是否需要准备带标注的案例发给学生？
   - 学生作业是否涉及文档标注提交？

#### **科研场景**

### Prompt 2

**User**：现在市面上有哪些文件标注系统？如果没有就直接说没有，如果有类似于文件标注功能的也可以介绍一下

市面上**确实存在**不少文件标注系统，根据不同的使用场景（AI训练、办公批注、团队协作、教学演示等），大致可以分为以下几类：

## 一、AI/机器学习训练用文档标注平台

这类工具主要用于为NLP（自然语言处理）、计算机视觉模型训练提供结构化标注数据，支持实体识别（NER）、文本分类、关系抽取等任务。

| 工具 | 特点 | 适用场景 |
|------|------|----------|
| **Label Studio** | 开源免费，支持多模态（文本、图像、音频、视频），支持LLM辅助预标注，有Python SDK和REST API  | 技术团队、需要自托管、跨领域AI项目 |
| **Doccano** | 开源轻量，专注于NLP任务（序列标注、文本分类），Docker一键部署，适合快速原型  | 学术研究、小型NLP项目、预算有限的团队 |
| **TagTog** | 内置PDF原生渲染，支持重叠实体、嵌套实体、关系标注，含主动学习功能，免费版限5,000标注/月  | 法律文档标注、复杂实体关系抽取 |
| **SuperAnnotate** | 企业级平台，支持多模态项目（文本+图像），有Agent Hub集成LLM自动标注，Token级精确选择  | 大规模企业项目、需要严格质量控制 |
| **Kili Technology** | 支持原生PDF渲染和OCR验证，具备SOC2、HIPAA合规认证，支持私有化部署  | 金融、医疗等受监管行业 |
| **Encord** | 支持多模态（文档、图像、视频、DICOM），集成GPT-4o/Gemini自动标注，具备完整数据管理  | 医疗影像、自动驾驶等复杂多模态AI |

## 二、PDF批注与文档编辑工具（办公场景）

这类是面向日常办公、文档审阅的PDF标注软件，主流选择包括：

- **福昕PDF编辑器**：功能全面，支持文本/图片修改、批注高亮、电子签章、文档加密、共享审阅
- **迅捷PDF编辑器**：界面类似Word，支持段落编辑、绘图标注、页面旋转、格式转换
- **轻闪PDF**：简洁易用，支持云端存储、OCR识别、多端同步（桌面+移动+浏览器）
- **转转大师PDF编辑器**：支持文字修改、图片插入、表格编辑、PDF拆分合并等功能
- **极速PDF编辑器**：专业级编辑工具，提供段落编辑、标注高亮、文档加密、拼写检查
- **极光PDF编辑器**：自主研发的PDF引擎，提供文件管理、注释、打印、文档保护等全生命周期功能
- **金舟PDF编辑器**：支持水印添加/去除、页面旋转、表单填写、批注修改

## 三、在线审阅与协作标注工具

面向团队审批流程，支持多人实时批注：

- **Filestage**：在线审阅工具，支持几乎所有常见文件格式的上下文标注，专注于创意内容审校流程，起步价€12/用户/月
- **Annotate**：文档协作工具，支持文本文件和PDF标注，可部署在私有云或本地服务器，有免费版

## 四、桌面/屏幕标注软件

- **Ink2Go**：桌面软件（$19.9一次性授权），可在屏幕任何内容（包括视频）上进行标注，保存为图片分享
- **PDF Annotator**：专为PDF设计（€69.95），支持评论、高亮、图片插入、手绘，可保存带标注的原文档或恢复 clean 版本

## 五、移动端标注App

- **Cubox**：个人阅读与标注工具，支持多格式导出（PDF、TXT、HTML），适合信息收集与管理
- **网课标注器/直播标注画笔**：屏幕画笔工具，用于网课讲解、直播标注，提供悬浮窗画笔功能

## 选择建议

- **如果你是AI研发团队**：首选 **Label Studio**（开源灵活）或 **Doccano**（轻量快速）；企业级需求考虑 **SuperAnnotate** 或 **Kili**（合规认证）
- **如果你是普通办公用户**：**福昕PDF编辑器** 或 **轻闪PDF** 功能全面且易上手
- **如果你需要团队审校流程**：**Filestage** 或 **Annotate** 更适合多人协作
- **如果你需要屏幕任意标注**：**Ink2Go** 可以在任何软件界面进行标注

总体而言，市面上的文件标注系统已经相当成熟，从免费开源到企业级商用，从纯文本到多模态AI训练，选择非常丰富。

# 裁判文书论证结构标记系统需求文档

## 1. 项目背景与目标

### 1.1 业务背景
本项目服务于法学研究中的**法律论证分析**，针对裁判文书（特别是"裁判理由"部分）进行论证结构的结构化标注与可视化。系统需支持研究团队对大量裁判文书进行形式化分析，提取论证模式，为后续自动化论证挖掘模型训练提供数据基础。

### 1.2 核心目标
- **论证挖掘**：从裁判文书中提取命题（Premise）与结论（Claim）
- **结构化表示**：将论证结构以标准化标签体系进行标注
- **可视化展示**：自动生成论证图示（Argumentation Diagram）
- **数据沉淀**：形成可复用的论证结构数据集，支持后续自动化推理与评估

### 1.3 应用场景
- 科研教学：分析法官裁判说理的基本模式与质量
- 实证研究：发现常见论证结构类型及潜在问题
- 律师辅助（未来）：针对法官论证习惯撰写针对性答辩意见

---

## 2. 功能需求

### 2.1 文本管理模块

#### 2.1.1 文本导入
- **支持格式**：PDF、纯文本、Word等至少3种常见格式
- **输入方式**：
  - 批量上传（管理员功能）
  - 单文件导入
  - 手动录入/复制粘贴（针对测试或小样本）
- **智能提取**：
  - 自动识别并提取"裁判理由"部分（针对参考案例/指导性案例有明确关键词；普通案例需支持规则匹配或人工校正）
  - 允许标注人员在界面上查看原文并手动调整提取范围
- **文档命名**：支持自定义命名，方便管理

#### 2.1.2 文本展示（左侧区域）
- **原文展示区**：
  - 保留原文格式，显示裁判文书内容
  - 支持在原文上直接进行标记（高亮或上下标），显示命题序号
  - 支持基于字符索引（Character Index）的定位
- **命题列表区**（左侧下半部分）：
  - **自动生成序号**：每个命题自动编号（1、2、3...）
  - **动态更新**：当命题合并、删除时，序号自动重新编排
  - **内容展示**：显示命题原文的一句话摘要
  - **手动编辑**：允许人工调整序号

### 2.2 标注核心模块（中间区域）

#### 2.2.1 标签体系管理
- **可配置性**：标签体系需支持动态调整，未来可增删改
- **一级标签（命题性质）**：
  1. 一般规范判断（General Normative Judgment）
  2. 个别事实判断（Individual Factual Judgment）
  3. 个别规范判断（Individual Normative Judgment）
  4. 一般事实判断（General Factual Judgment）
- **二级标签**（主要针对"一般规范判断"）：
  - 法律规范判断（Legal Normative Judgment）
  - 非法律规范判断（Non-legal Normative Judgment）：
    - 习惯规范
    - 道德规范
    - 行业惯例
  - 其他可扩展子类型
- **显示规则**：标注时需同时显示一级与二级标签（如"1. 一般规范判断 - 法律规范判断"），归属关系清晰

#### 2.2.2 命题标注（性质标注）
- **操作流程**：
  1. 在原文区选择文本片段（一句话或一段话）
  2. 在标签设置区选择对应的一级和二级标签
  3. 系统自动将命题加入左侧"命题列表"并分配序号
- **编辑功能**：支持修改标签、删除命题、合并命题（如将2、3号合并为2号）

#### 2.2.3 关系标注（论证关系）
- **关系类型**（当前5种，可扩展）：
  1. 支持（Support，标记为 S）
  2. 攻击/反对（Attack，标记为 A）
  3. 组合（Joint/Linked，多个命题共同支持结论）
  4. 其他（根据手册定义）
- **标注方式**：
  - 选择两个或多个已标注的命题
  - 选择关系类型
  - 系统自动生成符号表示（如 `S(1,2)` 表示1支持2）
- **复杂关系支持**：
  - **嵌套关系**：支持命题组之间的关系（如攻击一个支持关系）
  - **多对一关系**：多个命题组合后支持一个结论（如 `S({1,2},3)`）
  - 关系可被进一步标注关系（高阶关系）

### 2.3 可视化模块（右侧图示区）

#### 2.3.1 论证图示生成
- **自动渲染**：根据标注区的所有命题和关系，实时自动生成论证图示
- **图元标准**（需用户提供具体定义）：
  - 方框：表示单个命题（显示序号，不显示全文）
  - 箭头：表示支持/反对关系
  - 其他图形：表示组合、攻击等特定关系
- **布局算法**：自动排版，避免过于混乱，清晰展示论证结构
- **多图支持**：一个裁判文书可能包含多个独立论证结构，需支持分开展示

#### 2.3.2 图示交互
- **手动调整**：允许用户拖拽节点、调整布局
- **双向联动**：
  - 标注区修改 → 图示区自动更新
  - 图示区调整 → 同步更新标注数据
- **缩放与导航**：支持大图浏览、缩放、拖拽查看

### 2.4 协同标注模块（重点需求）

#### 2.4.1 多用户独立标注
- **任务分配**：
  - 管理员批量上传文档后，统一分配标注任务
  - 每份文档默认分配给**2人**（可配置）独立标注
  - 标注人员无需自己上传，点击"下一条"即可获取分配任务
- **数据隔离**：每个用户有独立的标注库，互不影响
- **角色权限**：
  - **标注员**：只能查看和编辑自己被分配的文档
  - **管理员/监督员**：可查看所有标注员的标注结果，进行对比和裁定

#### 2.4.2 差异比对与合并
- **自动比对**：系统对比同一文档的多个标注版本，自动识别差异
- **差异展示**：
  - 类似文档比对功能，高亮显示标注不一致的地方
  - 清晰展示：哪些命题标注不同、哪些关系缺失、标签选择差异
- **冲突解决**：
  - 管理员可查看差异详情
  - 支持编辑和裁定最终版本
  - 不一致处可提交专家裁决（线下讨论后线上确认）
- **一致性通过**：若多人标注完全一致，自动进入最终库

### 2.5 数据导出

#### 2.5.1 图示导出
- 论证图示支持导出为图片格式（PNG/JPG/SVG）

#### 2.5.2 标注数据导出
- **Excel格式**：包含命题列表、标签、关系矩阵等
- **JSON格式**：结构化数据，包含字符索引位置、标签层级、关系定义等，供模型训练使用
- **自定义命名**：支持导出文件命名

## 6. 风险与建议

### 6.1 潜在风险
- **需求复杂性**：嵌套关系、组合关系、协同比对等功能开发难度较高，建议分阶段交付（MVP先支持两人独立标注+简单支持关系，后续增加复杂关系与多人比对）

## 3. 具体需求

### 3.1 功能需求

#### 3.1.1 管理员功能需求

**FR-ADM-001：任务创建 [P0]**
- 描述：管理员应能创建新的标注任务
- 输入：任务名称、任务描述、截止日期、上传的裁判文书文件
- 处理：系统生成任务ID，存储文书内容至数据库
- 输出：任务创建成功确认，任务状态设为"待分配"

**FR-ADM-002：Schema配置 [P0]**
- 描述：配置该任务支持的标注类型与关系类型
- 输入：实体类型列表（如：主张、事实、理由、法条、结论）、关系类型列表（如：支持、反驳、补强）、颜色编码
- 处理：系统验证schema完整性（至少包含一个实体类型）
- 输出：Schema绑定至特定任务，标注者只能使用已配置的类型

**FR-ADM-003：人员分配 [P0]**
- 描述：为任务分配标注者和裁定者
- 输入：选择用户（多选）、指定每份文书的标注人数（>=2，用于冲突检测）、指定裁定者
- 处理：系统生成标注分配记录，发送通知
- 输出：分配确认，任务状态更新为"进行中"

**FR-ADM-004：进度监控 [P1]**
- 描述：查看任务整体进度与个体标注质量
- 输入：任务ID查询
- 处理：统计各标注者完成度、计算标注一致性率（Cohen's Kappa）
- 输出：仪表盘显示进度条、一致性热力图、导出统计报告

#### 3.1.2 标注者功能需求

**FR-ANN-001：任务接收 [P0]**
- 描述：查看并认领/接受分配的任务
- 输入：用户ID验证
- 处理：系统检索该用户的待办任务列表
- 输出：显示任务列表（包含优先级、截止日期、完成状态）

**FR-ANN-002：文本标注 [P0]**
- 描述：在裁判文书文本上执行Span标注和Relation标注
- 输入：鼠标选中文本、选择实体类型、添加标签注释
- 处理：
  - 记录实体位置（开始偏移量、结束偏移量）、类型、颜色
  - 支持重叠标注（同一文本可被多次标注不同类型）
  - 自动保存草稿（每30秒或手动触发）
- 输出：标注实体高亮显示，右侧边栏显示标注列表

**FR-ANN-003：关系建立 [P0]**
- 描述：建立标注实体间的论证关系
- 输入：选择源实体、目标实体、选择关系类型（支持、反驳、补强等）
- 处理：验证关系合法性（如：禁止自环）、存储关系数据
- 输出：可视化连线显示（不同颜色/线型表示不同关系）

**FR-ANN-004：标注提交 [P0]**
- 描述：完成标注后提交
- 输入：提交确认
- 处理：系统锁定该用户对此任务的编辑权限，触发冲突检测流程
- 输出：提交确认，状态更新为"待裁决"

**FR-ANN-005：标注修改 [P1]**
- 描述：在提交前或裁决前修改已有标注
- 输入：编辑或删除操作
- 处理：版本控制保留历史，支持撤销/重做（Undo/Redo栈深度>=10）
- 输出：实时更新界面显示

#### 3.1.3 裁定者功能需求

**FR-ARB-001：冲突查看 [P0]**
- 描述：查看系统检测到的标注冲突
- 输入：任务ID，冲突筛选条件（类型：实体冲突/关系冲突/缺失冲突）
- 处理：系统比对同一文书的多个标注版本，识别：
  - 实体边界不一致（重叠但不相等）
  - 实体类型分歧（同一文本标注为不同类型）
  - 关系连接差异
- 输出：分屏对比视图（左/中/右列显示不同标注者视角）

**FR-ARB-002：冲突裁决 [P0]**
- 描述：对冲突项做出裁定决策
- 输入：选择采纳哪个标注者的方案，或输入新的修正方案，添加裁决理由
- 处理：记录裁决结果，标记冲突状态为"已解决"
- 输出：生成黄金标准（Gold Standard）标注数据

**FR-ARB-003：批量裁决 [P1]**
- 描述：对相似冲突进行批量处理
- 输入：选择多个同类型冲突，应用统一决策规则
- 处理：系统应用规则，生成批量裁决记录
- 输出：更新后的黄金标准数据

#### 3.1.4 数据导出与可视化需求

**FR-EXP-001：数据导出 [P1]**
- 描述：导出标注结果
- 输入：选择任务、选择导出格式（JSON/CSV/XML）、选择数据版本（原始标注/裁决后）
- 处理：格式化数据，包含：
  - 元数据（任务信息、标注者、时间戳）
  - 实体列表（ID、文本、位置、类型）
  - 关系列表（源ID、目标ID、关系类型）
- 输出：下载文件，JSON Schema需符合JSON-LD标准

**FR-VIZ-001：论证结构图生成 [P1]**
- 描述：自动生成论证结构可视化图表
- 输入：标注数据（实体与关系）
- 处理：
  - 力导向图布局（Force-directed layout）
  - 节点按类型着色，边按关系类型定线型
  - 支持力证（实线绿色）、反驳（虚线红色）、补强（点线蓝色）
- 输出：交互式SVG图表，支持缩放、拖拽、点击查看详情

**FR-VIZ-002：图谱导出 [P2]**
- 描述：将生成的论证结构图导出为静态文件
- 输入：选择导出格式（PNG/SVG/PDF）、分辨率设置
- 处理：渲染图表，转换格式
- 输出：图片文件下载

## 三、盛浩宇

### Prompt 1

**User**："上面这几个文件是对方给的参考资料，但我不是学法律的，没看懂是什么意思。比如标签分为一级和二级，关系之间的处理是指什么？给我详细讲讲。"

**AI**：**核心概念类比：积木与连线**

1. 标签体系（积木块）：
   - **一级标签（大类）**：可以理解为命题的"身份"。比如是"事实判断"（发生了什么事）还是"规范判断"（法律怎么规定）。
   - **二级标签（细分）**：在身份下的具体属性。例如在"规范判断"下，是引用的"法律条文"还是"交易习惯"。
2. 论证关系（连线）：
   - 这是连接积木的逻辑胶水。系统核心包含 5 种关系：
     - **支持 (S)**：A 证明了 B 是对的。
     - **攻击 (A)**：A 反驳了 B。
     - **联合 (J)**：A 和 B 必须在一起才能支持 C。
     - **收敛 (C)**：A 支持 C，B 也支持 C，互不干扰。
     - **连串 (Ser)**：A 支持 B，B 又支持 C。
3. 难点提示：
   - 本项目的核心难点在于**"嵌套关系"**，即"关系"本身也可以成为被攻击或被支持的对象（例如：攻击"A支持B"这条逻辑线）。

### Prompt 2

**User**："项目有两个难点：复杂关系的嵌套，以及标注后自动绘图。我没有 Web 开发经验，不知道有没有现成工具（如 Drawio）能内嵌？另外多人标注不同时，系统怎么自动判断冲突？"

**AI**：**技术选型与算法建议**

1. 绘图工具选型：
   - 不建议直接内嵌 Drawio（过于重量级且难以数据绑定）。
   - **推荐方案**：使用数据驱动绘图库，如 **AntV G6** 或 **X6**，或者 **Cytoscape.js**。这些库支持将 JSON 数据直接渲染为拓扑图，且支持自定义节点（如"关系节点"）。
2. 嵌套关系实现：
   - 采用**"将线视为点"**的方案。在数据模型中，将"关系"也作为一个虚拟节点处理，这样就可以对"关系"进行连线。
3. 冲突检测算法：
   - **文本冲突**：基于字符索引（Start/End Index）。如果用户 A 标注了 [0, 10]，用户 B 标注了 [5, 15]，则判定为重叠冲突。
   - **逻辑冲突**：基于关系矩阵比对。比对两个用户对同一组命题建立的关系类型是否一致。

### Prompt 3

**User**："这是文档中的'待确认问题汇总'（18个问题），我不理解具体意思，给我讲讲。"

**AI**：**待确认问题深度解读**
这 18 个问题可以归纳为四个核心维度，建议按此逻辑与老师沟通：

1. 任务灵活性：
   - **Q1-Q3（改人/改数据/改标签）**：核心是确认任务创建后，是否允许"动态变更"。通常建议锁定标签体系，允许增加数据。
2. 标注体验：
   - **Q7-Q10（预览/序号/调图/撤销）**：关注标注员的效率。特别是"撤销重做"在图结构数据中实现难度较大，需确认是否必须。
3. 冲突裁定：

### Prompt 4

**User**："Word 文档里的撤销操作不是很常用吗？为什么你说在 Web 里面实现难度很大？"

**AI**：**Web 端实现 Undo/Redo 的复杂度分析**

1. **状态一致性**：Word 处理的是线性文本流，而本系统处理的是**图结构（Graph）**。一个操作可能同时涉及文本高亮、节点生成、连线变化。
2. **嵌套回滚噩梦**：如果用户删除了一个命题，那么连接该命题的所有"关系"、以及指向这些"关系"的"嵌套关系"都必须级联删除。撤销时，必须完美恢复这一整条链路，极易出错。
3. **内存管理**：如果采用"快照模式"保存每一步状态，数据量大时会导致浏览器内存溢出。
4. **建议方案**：采用**命令模式（Command Pattern）**，只记录操作指令（如 `create_node`, `delete_edge`），而非整个页面状态。

1. 多维标签筛选：
   - 除了按"进度"（未开始/进行中/已完成），建议增加"案由"（民事/刑事）、"难度"等业务标签。
2. 复合筛选栏：
   - 前端设计支持"且/或"逻辑的筛选器。
3. 批量动作：
   - 支持勾选多个案例后，进行"一键分发"、"批量导出"或"批量修改标签"。
4. 技术实现：
   - 后端 SQL 查询需优化，使用 `ORDER BY` 和索引，避免全表扫描导致页面卡顿。

### Prompt 6

**User**："老师希望导入文档后能直接修改原文内容（写裁判理由），好做吗？"

**AI**：**风险评估：数据一致性陷阱**

1. 核心风险：这是典型的"索引位移"问题。
   - 假设标注员在第 100 个字符处标注了一个命题。
   - 如果此时有人在第 50 个字符处插入了一句话，原文长度变了，原本第 100 个字符的标注就会"错位"，指向错误的文字。
2. 解决方案建议：
   - **预处理校对模式**：建议将"修改原文"作为一个独立的前置阶段。只有在原文被"锁定"后，才允许开始标注。
3. 沟通话术：
   - "老师，为了保证标注数据的精准定位，我们需要将'文本清洗'和'逻辑标注'分开。如果边标边改，之前的标注位置可能会乱掉，导致数据作废。"

## 一、核心规则：使用 <<include>>（包含关系）

**定义**：包含关系表示**基础用例（Base Use Case）** 在执行过程中**必然**调用**包含用例（Included Use Case）** 的行为。包含用例是基础用例的一个**强制性、完整子步骤**，不可省略。

**符号**：带虚线和箭头的连线，箭头指向被包含的用例，标记 <<include>>。

**适用条件**：

- 子用例被多个父用例共享 -> 抽取为独立用例，用 include 复用。
- 子用例虽然是父用例的一部分，但具有**独立的业务价值**（例如"上传文件"本身是一个完整的用户可理解的功能）。
- 如果子用例仅仅是一个原子操作（如"点击按钮"），没有独立意义，则**不应**建模为独立用例，而应写在基本流的步骤中。

**分析**：

- **配置标签体系**：独立价值吗？是的，任务创建者可能在不同任务中重复使用同一套标签配置（比如复制上次的配置），或者单独预览标签体系。可以建模为独立用例。
- **上传裁判文书**：独立价值吗？是的，可以在创建任务前或任务中上传多份文书，也可以批量上传。独立用例有意义。
- **创建标注任务**：必然要执行配置标签体系和上传文书（否则任务无效）。因此使用 <<include>>。

## 三、其他关系的适用场景（对比）

| 关系类型 | 关键字 | 含义  | 示例  |
| --- | --- | --- | --- |
| **包含（include）** | <<include>> | 基础用例**必须**执行子用例 | 创建任务 -> 必须配置标签、上传文书 |
| **扩展（extend）** | <<extend>> | 子用例**可选**，在特定条件下触发 | 标注时 -> 如果发现原文错误，可扩展"申请修改原文" |
| **泛化（generalization）** | ------（空心三角箭头） | 子用例是父用例的特殊化 | "导出Excel"和"导出JSON"都泛化为"导出数据" |

**规则总结**：

- **必选且完整子功能** -> <<include>>
- **可选或异常分支** -> <<extend>>
- **同一业务目标的不同实现方式** -> 泛化

## 一、为什么不作为独立用例？

### 1. 不符合"独立业务价值"原则

独立用例应该是一个**用户可独立发起、具有明确业务目标**的操作。例如"上传裁判文书"本身是一个完整目标（用户想导入素材）。

"对已上传的文书内容进行修正"不是一个独立目标，它**依附于**"上传裁判文书"或"编辑任务素材"这类父功能。用户不会单独登录系统只为了"修正提取边界"，而是在查看或修改已有任务时作为辅助操作出现。

### 2. 符合 UML 扩展关系（<<extend>>）的典型场景

扩展关系用于描述**可选、有条件触发**的行为，并且该行为是对基础用例的补充，而非必选步骤。

- 基础用例："上传裁判文书"
- 扩展条件：系统自动提取的"裁判理由/本院认为"范围不准确
- 扩展行为：用户手动修正提取边界

这正好是 <<extend>> 的经典用法。

## 二、推荐做法：使用 <<extend>> 表示"修正提取边界"

### 详细用例描述中的写法（在"上传裁判文书"用例中）

**替代流**（或扩展点）中写明：

**替代流 A1：系统自动提取范围不准确**

1. 系统完成裁判理由/本院认为的自动提取后，在界面上以虚线框标出提取的文本范围。
2. 用户发现范围错误（多选/漏选），点击"修正范围"按钮。
3. 系统允许用户在原文中重新拖拽选择正确的起始和结束位置。
4. 用户确认后，系统更新提取后的文本，用于后续标注。
5. （可选）系统记录用户修正行为，用于后续优化提取算法。

**后置条件**中可以补充：

- 若用户未修正，则使用系统默认提取范围；若用户已修正，则保存修正后的范围。

## 三、什么情况下需要详细描述？

### 需要详细描述的场景

- 该功能**逻辑复杂**（例如支持多版本对比修正、多人协同修正、权限审批等）。
- 该功能**对后续流程有重大影响**（例如修正后会导致已有命题编号错位、关系失效等，需要特殊处理）。
- 用户调研或需求中**明确强调**这是一个高频、关键的操作。

### 不需要单独建模为用例的场景
