## 一、开场：用例选取与价值说明（约 1 分钟）

> **讲稿：**
>
> 各位老师好。接下来我们选取系统中最能体现业务价值的一条完整用例——**「双人背对背标注与冲突裁定全流程」**，串联展示本项目从需求到部署的工程过程。

> 这条用例覆盖四个角色、三个阶段状态流转，以及系统最核心的标注与裁定能力：

## 二、需求分析阶段（约 2 分钟）

> **讲稿：**
>
> 需求阶段，我们从法学研究场景出发，明确了系统的核心问题：**如何对裁判文书「裁判理由」进行结构化标注，并在多人独立标注后通过人工裁定形成权威数据集。**
>
> 在 P1 阶段，我们完成了用户调研、用户故事编写和需求规格说明书（SRS）。针对本次用例，需求上重点落实了以下几类约束：

### 2.1 业务规则（来自 SRS）

| 规则 | 说明 | 需求来源 |
| --- | --- | --- |
| 四阶段不可逆流程 | 创建 → 标注 → 冲突解决 → 结果输出 | SRS §1.3、§2.4 |
| 背对背标注 | 同一文书至少两名标注员，提交前互不可见 | US-02、US-04 |
| 唯一裁定者 | 每个任务指定一名裁定者，一次裁定生效 | SRS §2.4 |
| 标签体系锁定 | 任务启动后指南版本不可修改 | US-03 |
| 命题序号规则 | 按文本出现顺序编号，删除后自动前移 | FR-008 |
| 结果导出 | 命题/关系 CSV + 论证图 PNG，ZIP 打包 | US-12、US-13 |

### 2.2 本用例涉及的功能需求编号

| 阶段 | 关键 FR | 内容摘要 |
| --- | --- | --- |
| 创建任务 | FR-005、FR-006 | 选取/上传文书，分配标注员与裁定者 |
| 标注 | FR-007～FR-015 | 三栏界面、划词命题、标签、关系、图示 |
| 裁定 | FR-022～FR-024 | 双标注对比、全部采纳、部分修改 |
| 导出 | FR-025～FR-027 | 结果查看、CSV/PNG/ZIP 导出 |

## 三、总体结构设计阶段（约 2 分钟）

> **讲稿：**
>
> 进入 P2 架构设计阶段，我们面对的核心问题是：**4 人团队、10 周工期、无 Web 开发经验，如何在保证并行开发的同时降低协作冲突。**
>
> 最终采用 **模块化垂直架构 + 前后端分离**，按业务功能划分为四个模块，每人负责一个模块的前后端与数据库设计：

### 3.1 架构分层与模块划分

```
┌─────────────────────────────────────────┐
│           前端展示层（Vue 3 + Vite）      │
├─────────────────────────────────────────┤
│         后端服务层（Spring Boot 3）       │
│  ┌────────┬────────┬────────┬──────────┐ │
│  │用户权限│任务管理│标注核心│冲突裁定  │ │
│  │  模块  │  模块  │  模块  │与导出模块│ │
│  └────────┴────────┴────────┴──────────┘ │
├─────────────────────────────────────────┤
│           数据持久层（MySQL 8）            │
└─────────────────────────────────────────┘
```

| 模块 | 本用例中的职责 |
| --- | --- |
| 用户权限 | JWT 登录、角色路由守卫（admin / creator / user） |
| 任务管理 | 任务 CRUD、成员分配、四阶段状态机 |
| 标注核心 | 命题/关系标注、论证图生成与布局保存 |
| 冲突裁定与导出 | 双标注对比、采纳/修改、前端 ZIP 导出 |

### 3.2 关键架构决策（ADR）

| ADR | 决策 | 理由 |
| --- | --- | --- |
| ADR-001 | 模块化垂直架构 | 并行开发、责任清晰、减少 Git 冲突 |
| ADR-002 | Vue 3 + Spring Boot 分离 | 交互复杂的前端 + 成熟的后端生态 |
| ADR-003 | MySQL 单库 | 数据量与并发满足课程场景，运维简单 |
| ADR-004 | JWT 无状态认证 | 前后端分离天然适配，跨域友好 |

> **讲稿（衔接）：**
>
> 需要说明的是，P2 设计稿与最终实现存在少量差异——例如 API 路径为 `/api/...` 而非 `/api/v1/...`，JWT 采用自研拦截器而非 Spring Security。这些差异已在 `docs/P2/架构设计文档.md` 末尾「与当前实现的差异说明」中记录，**架构方向与模块划分保持一致**。

## 四、详细设计阶段（约 3 分钟）

> **讲稿：**
>
> P3 详细设计阶段，我们将架构模块落地为可实现的类图、API 规范与数据库 schema。针对本用例，重点说明三条设计主线：**数据模型、接口契约、前端页面流转**。

### 4.1 数据模型：支撑「标注 → 裁定 → 导出」

本用例涉及的核心实体关系（摘自 ER 图）：

| 实体 | 作用 |
| --- | --- |
| `task` / `task_member` | 任务及标注员、裁定者分配 |
| `task_document` | 任务内文书，承载阶段状态 |
| `annotation` + `proposition` + `argument_relation` | 规范化存储标注结果（非 JSON 快照） |
| `arbitration_snapshot` | 裁定元数据，每文书一条最终记录 |
| `guide_version` + `label_l1/l2` + `relation_type` | 任务绑定的标签与关系体系 |

**设计演进要点**（初版 → 实现）：

- 标注数据由 JSON 快照改为 **proposition / relation_member 规范化表**，便于裁定时的差异比对；
- 导出由后端文件流改为 **前端浏览器端 ZIP 打包**，降低后端复杂度；
- 新增 **message 表** 支持任务阶段变更通知。

> **展示建议：** 打开 `docs/img/P3-ER图.png` 或 `docs/img/P3-实体间关系.png`，指出 task → annotation → proposition 链路。

### 4.2 接口设计：本用例关键 API

| 阶段 | 方法 | 路径 | 说明 |
| --- | --- | --- | --- |
| 登录 | POST | `/api/auth/login` | 获取 JWT |
| 创建任务 | POST | `/api/tasks` | 选文书、指南、成员 |
| 读取标注 | GET | `/api/tasks/{taskId}/items/{dataId}` | 加载文书与已有标注 |
| 提交标注 | POST | `/api/annotations/submit` | 暂存/正式提交，触发阶段推进 |
| 保存布局 | POST | `/api/annotations/layout` | 论证图节点位置 |
| 读取裁定 | GET | `/api/reviews/{taskId}` | 对比多标注员结果 |
| 全部采纳 | POST | `/api/reviews/adopt` | 一键采纳某版本 |
| 部分修改 | POST | `/api/reviews/manual` | 裁定草稿提交与确认 |

统一响应格式：`{ "code": 200, "message": "success", "data": {} }`

```text
