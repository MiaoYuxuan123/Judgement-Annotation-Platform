## 二、AI 助力分析（Q1）

### 1. 核心助力点

- **效率提升**：AI 能在短时间内输出类图框架、API 文档模板、建表 SQL 初稿，把团队从重复格式编写中解放出来，让我们把时间集中在**业务逻辑、权限划分、流程合理性**等核心设计判断上，而不是基础结构搭建。
- **规范补齐**：AI 自动提供 SOLID 原则、设计模式、第三范式、RESTful 风格、统一响应体、错误码体系等工程规范，弥补了人工设计时容易忽略的细节，让整体设计更标准、更易落地。
- **风险前置**：AI 可批量枚举潜在设计缺陷，包括接口未鉴权、参数缺失、索引不合理、数据冗余、权限未隔离等问题，帮助团队在设计阶段就提前发现风险，避免开发阶段返工。
- **知识辅助**：在设计模式选型、SOLID 解读、数据库范式判断、接口规范定义等理论点上，AI 提供了标准化参考，降低了团队成员的学习与沟通成本。

### 2. 助力效果量化

- 基础设计初稿整体完成效率提升约 **60%**，类图、API、数据库初稿在 1 小时内完成。
- AI 提出的 **12 条设计优化建议被采纳**，包括统一错误码体系、高频字段建立索引、接口权限分离、标注流程拆分等。
- 在 AI 辅助下，团队提前识别 **5 个未考虑到的设计风险**：里氏替换原则违规、多对多关系未拆表、接口过于臃肿、密码明文存储风险、角色权限未隔离。

## 三、AI 误导分析（Q2）

### 1. 量化数据

SOLID 检查中发现的 AI 设计问题数量：**5 处**

### 2. 核心误导点

#### （1）类图设计维度

- **最严重问题**：AI 生成的 `TaskService` 类同时承担**任务创建、任务分配、标注处理、裁定逻辑、结果统计、文件导出**六项职责，完全违背**单一职责原则**，导致类体积庞大、难以维护、难以测试。
- 其次，AI 设计 `Admin`、`Annotator`、`Reviewer` 继承自 `User` 父类，但子类重写了权限判断与流程方法，**无法实现里氏替换**，子类不能安全替换父类使用。
- 其他问题：设计 “万能接口” 包含大量无关方法，违反**接口隔离原则**；高层业务逻辑直接依赖数据库实现类，违反**依赖倒转原则**。

#### （2）API 设计维度

- AI 生成的接口命名不规范，使用 `getTaskList`、`updateUserInfo` 等 RPC 风格，而非标准 RESTful 风格。
- 未区分**管理员、标注员、裁定者**的接口权限，所有接口公开访问，存在严重安全漏洞。
- 几乎所有接口**缺少 Token 认证**、缺少角色校验。
- 参数校验仅判断是否为空，未做格式、长度、范围、合法性校验。
- 错误码体系混乱，无统一规划，前端无法统一处理异常。

#### （3）数据库设计维度

- 部分表结构违反第三范式，存在大量冗余字段，例如任务表中重复存储标注员信息。
- 高频查询字段如 `user_id`、`task_id`、`document_id` **未建立索引**，会导致后期查询性能极差。
- 安全设计严重缺失：建议密码**明文存储**，未做任何加密处理；手机号等隐私数据未脱敏。
- 未考虑法律文书系统**数据可追溯、操作留痕**的需求，缺少创建时间、更新时间、操作人字段。

### 3. 误导原因分析

- **业务理解不足**：AI 只按通用系统逻辑生成设计，完全不理解法律标注系统**多角色、强流程、权限严格分离、数据可审计**的核心业务特点。
- **工程落地缺失**：AI 只满足理论规范，不考虑真实系统的**性能、安全、可维护性、前后端协作成本**。
- **原则权衡能力缺失**：AI 只会机械套用单一原则（如第三范式），不会在**规范、性能、开发成本、查询效率**之间做合理权衡。
- **流程认知不足**：AI 不理解 “任务→分配→标注→裁定→导出” 这一专业法律标注流程，导致流程节点设计混乱、职责交叉。

## 四、Prompt 改进计划（Q3）

### 1. 上阶段改进措施的验证结果

- **上阶段优化方向**：在 Prompt 中增加业务约束、明确输出格式、强调工程规范。
- **验证结果**：AI 输出的**格式规范性明显提升**，结构更清晰；但**业务贴合度仍然严重不足**，深层逻辑错误依然大量存在。约束细化后，表面违规点减少约 40%，但核心设计错误仍需人工全面校验与修正。

### 2. 本阶段新的改进措施

|   优化方向   |             原 Prompt 问题              |                    改进后 Prompt 设计思路                    |
| :----------: | :-------------------------------------: | :----------------------------------------------------------: |
|   类图生成   | 仅要求功能覆盖，未强制 SOLID 与职责拆分 | 明确要求：必须严格遵循 SOLID；按职责拆分为用户 / 任务 / 标注 / 裁定 / 文书 / 配置六大服务类；禁止大类；输出时附带 SOLID 自检说明 |
| API 文档生成 |   未明确权限、校验、错误码、安全规则    | 强制 RESTful 风格；区分管理员 / 标注员 / 裁定者权限；所有接口必须带 Token 认证；错误码按模块编号；参数必须包含格式、范围、权限校验 |
|  数据库设计  |   未强调隐私安全、性能、索引、可追溯    | 必须满足第三范式；密码必须 BCrypt 哈希存储；手机号脱敏；`user_id`/`task_id`/`document_id` 必须建索引；必须包含创建 / 更新时间与操作人字段 |
|   通用输出   |         无自检步骤，输出不可靠          | 要求 AI 输出后必须附带自检清单：SOLID 检查、范式检查、安全检查、性能检查、权限检查 |

## 五、本阶段的核心工程判断

### 1. 决策内容

- **类图设计**：按单一职责原则，拆分为 `UserService`、`TaskService`、`AnnotationService`、`ReviewService`、`DocumentService`、`ConfigService` 六大独立服务；采用**策略模式**处理不同标注规则的切换，采用**工厂模式**创建不同角色对象，保证扩展性与维护性。
- **API 设计**：全面采用 RESTful 风格；统一错误码体系（100xx 用户、200xx 任务、300xx 标注、400xx 裁定）；所有接口强制 Token 认证与角色校验；完善参数格式、长度、范围、权限四层校验。
- **数据库设计**：严格满足第三范式；`user_id`、`task_id`、`document_id` 建立索引；密码使用 BCrypt 哈希加盐存储；手机号脱敏存储；任务表保留少量状态冗余字段以提升查询性能；所有表增加操作日志与时间戳字段，满足法律系统可追溯要求。

### 2. 为什么 AI 无法做这个决策

- **业务特殊性无法理解**：AI 不理解法律文书系统**流程严谨、权限严格、数据安全、操作可追溯、结果可导出**的强行业要求。
- **原则权衡能力缺失**：AI 只会机械遵守单一规范，无法在**范式、性能、开发成本、维护成本**之间做出工程化折中判断。
- **工程经验不足**：AI 无法预判长期风险，如接口过于臃肿导致维护困难、索引缺失导致性能崩溃、权限混乱导致安全漏洞。
- **协作场景不理解**：AI 不了解前后端协作习惯、团队技术栈特点、接口对接规范，无法输出真正可落地的工程设计。
